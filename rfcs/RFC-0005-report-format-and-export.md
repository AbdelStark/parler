# RFC-0005: Report Format and Export Integrations

**Status**: Draft  
**Date**: 2026-04-09  
**Repo**: parler

---

## Abstract

This RFC defines the report formats produced by `parler` and the export integrations with external task management and collaboration tools. It specifies the Markdown, HTML, and JSON formats and the data contracts for Notion, Linear, Jira, and Slack exports.

---

## Motivation

The Decision Log is only useful if it gets to the people who need to act on it. A perfect decision log that sits in a local file on the organizer's laptop solves nothing.

`parler` must make it trivially easy to:
1. Share the decision log with meeting participants (copy-paste into Slack, email, Notion)
2. Create actionable tasks from commitments (Linear issues, Jira tickets)
3. Archive decisions in a searchable format (Notion database, Confluence)

The format design must optimize for "this looks good when someone shares a screenshot of it" — viral shareability is part of the tool's growth strategy.

---

## Markdown Format (default)

The default output format. Designed to render well in:
- GitHub issues and PRs
- Notion pages
- Confluence pages
- Slack markdown (with minor adjustments)
- Any Markdown preview tool

## HTML Format

A self-contained HTML file with:
- Inline CSS (no external dependencies — shares cleanly as a file attachment)
- Clean, professional design (suitable for screenshot-sharing)
- A timeline view showing when in the meeting each decision happened
- Collapsible transcript excerpts
- Print-ready (generates clean PDF if printed)

### Timeline view

The HTML report includes a visual meeting timeline:

```
0:00 ─────────────┬──────────────────────────────────────────────── 47:23
                  │
  14:02 ─────────●─── D1: Launch date set to May 15
  22:41 ──────────────────────●─── D2: Switch to Mistral Small
  29:44 ──────────────────────────────●─── Q1: GPU budget open
  31:22 ────────────────────────────────────●─── R1: GPT-4o rejected
  38:15 ──────────────────────────────────────────●─── D3: Monthly review
  44:10 ──────────────────────────────────────────────────●─── Q2: OSS harness
```

Each item on the timeline is clickable and jumps to the relevant section of the report.

### Design tokens

The HTML template uses a minimal design:
- Font: system-ui (no web fonts, clean on all platforms)
- Colors: `#1a1a2e` (heading), `#16213e` (text), `#0f3460` (accent), `#533483` (decision green variant)
- Monospace for timestamps and quotes
- Responsive (works on mobile)

## JSON Format

The JSON output is the machine-readable `DecisionLog` schema from RFC-0003, with one addition: the full transcript embedded under a `transcript` key if `--include-transcript` is specified.

```json
{
  "schema_version": "0.1.0",
  "generated_at": "2026-04-09T14:45:00Z",
  "audio": {
    "filename": "meeting.mp3",
    "duration_s": 2843,
    "languages": ["fr", "en"]
  },
  "decisions": [...],
  "commitments": [...],
  "rejected": [...],
  "open_questions": [...],
  "metadata": {...}
}
```

## Export Integrations

### Notion export

Creates one Notion database entry per session, with sub-pages for decisions and commitments:

```python
from parler.export import NotionExporter

exporter = NotionExporter(
    api_key=os.environ["NOTION_API_KEY"],
    database_id="your-database-id"
)
exporter.export(decision_log)
```

Notion structure:
```
Meeting: 2026-04-09 Product Sync
├── Properties: Date, Duration, Languages, Participants
├── Decisions
│   ├── D1: Launch date set to May 15 [Decision]
│   └── ...
├── Commitments
│   ├── C1: Sophie → Review deployment checklist → 2026-04-12 [Task]
│   └── ...
└── Open Questions
    ├── Q1: GPU budget approval [Question]
    └── ...
```

### Linear export

Creates one Linear issue per commitment, assigned to the named owner, with the deadline as the due date:

```python
from parler.export import LinearExporter

exporter = LinearExporter(
    api_key=os.environ["LINEAR_API_KEY"],
    team_id="your-team-id"
)
exporter.export(decision_log)
```

Linear issue format:
```
Title: [Meeting 2026-04-09] Run benchmarks on Mistral Small
Assignee: Thomas (matched by name from Linear team members)
Due date: 2026-04-11
Description:
  From the product sync on April 9.
  
  Context: "The cost reduction justifies the quality tradeoff for this use case"
  
  Related decision: D2 — Switch to Mistral Small for classification pipeline
  
  Generated by parler
```

Name matching: Linear export uses fuzzy matching of speaker names to Linear team member names. Unmatched names are left unassigned with a note.

### Jira export

Similar to Linear, creates Jira issues per commitment:

```python
from parler.export import JiraExporter

exporter = JiraExporter(
    server_url="https://your-company.atlassian.net",
    email=os.environ["JIRA_EMAIL"],
    api_token=os.environ["JIRA_API_TOKEN"],
    project_key="PROJ"
)
exporter.export(decision_log)
```

### Slack export

Sends a formatted message to a Slack channel via webhook:

```python
from parler.export import SlackExporter

exporter = SlackExporter(webhook_url=os.environ["SLACK_WEBHOOK_URL"])
exporter.export(decision_log)
```

Slack message format uses Slack's Block Kit for rich formatting:
- Summary card at top (3 decisions, 5 commitments, 2 open questions)
- Expandable sections for each category
- @mentions for commitment owners (if Slack user IDs are available via `--slack-users` config)

## Viral Format: The Screenshot

The HTML report is designed for the specific use case of "posting a screenshot on Twitter."

Key design decisions for shareability:
1. **Compact summary card at the top**: the first screen should tell the whole story
2. **Numbers that make you go "wow"**: "47 minutes → 3 decisions, 5 commitments, 2 open questions"
3. **Quote excerpts**: showing the French/English quote alongside the extracted decision makes it feel real, not synthetic
4. **"Generated by parler" credit**: visible in the output, provides discovery path

The canonical demo screenshot:
1. Input: a 30-minute French/English meeting recording
2. Output: the HTML report
3. Screenshot: the summary card + timeline view + first 3 decisions

Posted on Twitter: "30 minutes of meeting in French/English → clean decision log in 60 seconds, using Voxtral + Mistral. Completely free on Mistral's API. Try it: [repo link]"

This format has been validated in similar contexts (the "I migrated X to Y" format on developer Twitter). The French-first posting makes it novel for the English-speaking AI community and authentic for the French tech community.

## Open Questions

1. **Calendar integration**: should `parler` support sending the decision log to meeting participants via calendar integration (Google Calendar, Outlook)? After the meeting, automatically send the decision log to all invited attendees? This requires OAuth and calendar API access — significant additional scope.

2. **Incremental export**: if the same meeting is processed multiple times (e.g., the transcript was low quality and was re-run after audio enhancement), should the export integrations update existing Notion entries / Linear issues rather than creating duplicates? This requires a session ID → external entity ID mapping stored locally.

3. **Custom templates**: power users may want custom report templates (company branding, different structure, additional sections). Should `parler` support a `--template` flag pointing to a Jinja2 template file?
